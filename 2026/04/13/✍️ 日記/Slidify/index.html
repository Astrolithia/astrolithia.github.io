用户登录模块实现

  • 你的项目是前后端分离单体，不是微服务，没必要一上来就做 Gateway + JWT。
  • 你本身已经要用 Redis 做任务队列、限流、缓存，顺手把登录态也放 Redis，工程复杂度最低。
  • AI 平台很需要“踢人下线、禁用账号、限制并发、多端控制、按用户计次”，这些都是 Sa-Token + Redis 的强项。
  • 如果你用纯 JWT，后面一旦要做登出失效、封号立即生效、强制下线、续签、黑名单，通常还是得把 Redis 再补回来，复杂度反而更高。
  • Spring Session + Redis 更偏传统 Web Session/Cookie 方案，不是最适合你的 SPA + API 架构。
  • Spring Security + JWT 双 Token 当然也能做，但对你这个 MVP 来说偏重，尤其你现在核心难点其实在 AI 任务、SSE、导出，不在认证本身。
  • 登录：账号密码登录，密码 Bcrypt
  • 鉴权：Sa-Token 登录态存在 Redis
  • 权限：USER / ADMIN 两级 RBAC 先够用
  • 会话：支持自动续签、管理员踢下线
  • 限制：同账号最大在线设备数可以后补
  • SSE：不要默认走自定义 Header 的原生 EventSource
  • SSE 更稳的做法是：HttpOnly Cookie 同域鉴权，或者前端改用支持 Header 的 SSE/Fetch 方案
方案 核心思路 适用场景 优点 缺点 / 风险 当前主流度
Session + Cookie 服务端保存会话，浏览器带 Cookie 单体应用、SSR、后台管理系统 简单、成熟、服务端可控、登出立即生效 扩容后要做会话共享；前后端分离跨域时麻烦 很主流
Spring Session + Redis 用 Redis 统一存 HttpSession 集群部署的单体/中型系统 保留 Session 简单模型，同时支持集群共享、并发会话控制 仍然是有状态；跨域和移动端体验不如 Token 灵活 很主流
JWT Access Token + Refresh Token Access Token 短期访问，Refresh Token 续期 前后端分离、APP、小程序、开放 API 前后端分离友好、接口调用方便、跨服务传递容易 真做登出、踢下线、封号、黑名单后，通常还是要 Redis；实现复杂度高于想象 很主流
OAuth2 / OIDC 认证交给授权服务器，业务系统只做 Client / Resource Server SSO、第三方登录、企业统一身份、微服务 标准化最强，适合多系统、组织级认证、第三方接入 架构最重，落地成本高，MVP 不划算 企业级主流
Gateway + JWT / OIDC 认证在网关统一校验，再把用户上下文传给下游 微服务、API 网关架构 鉴权集中、下游服务更干净、统一安全策略 只适合微服务；单体项目上这个会过度设计 微服务场景主流
Sa-Token + Redis 轻量 Token/会话框架，状态存在 Redis

Vue 的组件风格

选项式 API 和组合式 API
选项式 API

特点：

  • 数据写在 data
  • 方法写在 methods
  • 计算属性写在 computed
  • 生命周期写在 mounted

也就是说，同一个功能会被拆散到不同区域。

组合式 API

特点：

  • 跟“计数器”相关的状态、方法、计算属性都写在一起
  • 不再依赖 this
  • 更适合逻辑复用和大型组件维护

选项式 API
按 Vue 配置项分类：

  • data
  • methods
  • computed
  • watch

组合式 API
按业务逻辑组织：

  • 用户相关逻辑放一起
  • 搜索相关逻辑放一起
  • 上传相关逻辑放一起

:key="item.path" :是 v-bind: 的简写
key 是 vue 自己用的，这个列表项的唯一身份是谁
可以理解为元素的身份证号
为什么要设 key：

  • Vue 在更新列表时，需要知道哪个旧节点对应哪个新节点
  • 如果没有 key,可能用复用错 DOM
  • 一旦以后菜单是动态增删、排序变化，没 key 可能会出现问题
    :index 给每个菜单项一个菜单内部标识
  • el-menu 靠它判断当前哪个菜单项是激活状态
  • 又加了 router 所以 index 还会被当作跳转目标
  • 把 index 设为目标路由后 点击这个菜单项就能跳到对应页面
