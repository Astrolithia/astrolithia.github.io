用户登录模块实现

  • 你的项目是前后端分离单体，不是微服务，没必要一上来就做 Gateway + JWT。
  • 你本身已经要用 Redis 做任务队列、限流、缓存，顺手把登录态也放 Redis，工程复杂度最低。
  • AI 平台很需要“踢人下线、禁用账号、限制并发、多端控制、按用户计次”，这些都是 Sa-Token + Redis 的强项。
  • 如果你用纯 JWT，后面一旦要做登出失效、封号立即生效、强制下线、续签、黑名单，通常还是得把 Redis 再补回来，复杂度反而更高。
  • Spring Session + Redis 更偏传统 Web Session/Cookie 方案，不是最适合你的 SPA + API 架构。
  • Spring Security + JWT 双 Token 当然也能做，但对你这个 MVP 来说偏重，尤其你现在核心难点其实在 AI 任务、SSE、导出，不在认证本身。
  • 登录：账号密码登录，密码 Bcrypt
  • 鉴权：Sa-Token 登录态存在 Redis
  • 权限：USER / ADMIN 两级 RBAC 先够用
  • 会话：支持自动续签、管理员踢下线
  • 限制：同账号最大在线设备数可以后补
  • SSE：不要默认走自定义 Header 的原生 EventSource
  • SSE 更稳的做法是：HttpOnly Cookie 同域鉴权，或者前端改用支持 Header 的 SSE/Fetch 方案
方案 核心思路 适用场景 优点 缺点 / 风险 当前主流度
Session + Cookie 服务端保存会话，浏览器带 Cookie 单体应用、SSR、后台管理系统 简单、成熟、服务端可控、登出立即生效 扩容后要做会话共享；前后端分离跨域时麻烦 很主流
Spring Session + Redis 用 Redis 统一存 HttpSession 集群部署的单体/中型系统 保留 Session 简单模型，同时支持集群共享、并发会话控制 仍然是有状态；跨域和移动端体验不如 Token 灵活 很主流
JWT Access Token + Refresh Token Access Token 短期访问，Refresh Token 续期 前后端分离、APP、小程序、开放 API 前后端分离友好、接口调用方便、跨服务传递容易 真做登出、踢下线、封号、黑名单后，通常还是要 Redis；实现复杂度高于想象 很主流
OAuth2 / OIDC 认证交给授权服务器，业务系统只做 Client / Resource Server SSO、第三方登录、企业统一身份、微服务 标准化最强，适合多系统、组织级认证、第三方接入 架构最重，落地成本高，MVP 不划算 企业级主流
Gateway + JWT / OIDC 认证在网关统一校验，再把用户上下文传给下游 微服务、API 网关架构 鉴权集中、下游服务更干净、统一安全策略 只适合微服务；单体项目上这个会过度设计 微服务场景主流
Sa-Token + Redis 轻量 Token/会话框架，状态存在 Redis

Vue 的组件风格

选项式 API 和组合式 API
选项式 API

特点：

  • 数据写在 data
  • 方法写在 methods
  • 计算属性写在 computed
  • 生命周期写在 mounted

也就是说，同一个功能会被拆散到不同区域。

组合式 API

特点：

  • 跟“计数器”相关的状态、方法、计算属性都写在一起
  • 不再依赖 this
  • 更适合逻辑复用和大型组件维护

选项式 API
按 Vue 配置项分类：

  • data
  • methods
  • computed
  • watch

组合式 API
按业务逻辑组织：

  • 用户相关逻辑放一起
  • 搜索相关逻辑放一起
  • 上传相关逻辑放一起

@Resource 作用是把 Spring 容器里的对象注入进来
意思就是：

  • 你自己不用 new SysUserServiceImpl()
  • Spring 会帮你找到 SysUserService 对应的实现类
  • 然后把它注入到 sysUserService 这个字段里
    放在这里的作用就是：
  • Controller 想调用用户注册逻辑
  • 但注册逻辑不应该写在 Controller 里
  • 所以 Controller 通过注入 Service 来调用业务层

Controller 负责接请求
Service 负责处理业务
@Resource 负责把 Service 接进来

@RestController
它的作用是：告诉 Spring，这个类是一个接口控制器，并且方法返回值直接作为 HTTP 响应体返回。如果类上有 @RestController，Spring 就会把这个返回值自动转成 JSON 返回给前端。

@RequestMapping(“/user”)
它的作用是：给整个 Controller 统一加一个访问路径前缀。

关于 @Autowired：

  • 它的作用和 @Resource 一样，都是做“依赖注入”
  • 也就是：让 Spring 自动把 SysUserService 这样的对象塞进 Controller / Service 里
  • @Autowired 是 Spring 自己的注解，核心是“按类型注入”
  • @Resource 是 Jakarta 标准注解，通常理解成“优先按名称，再按类型”

PPT 编辑加上乐观锁，原因：防止冲突

用户 A 打开编辑页，读取到 version = 3，开始写标题
用户 A 在手机上同时也打开了这个作品，也读到 version = 3

用户 A 电脑端保存 → version 变成 4
用户 A 手机端也点保存 → 带着 version = 3 来更新
→ WHERE version = 3 匹配不到（已经是4了）→ 更新失败
→ 返回”内容已被修改，请刷新后重试”
如果没有乐观锁，手机端的旧内容会直接覆盖电脑端刚保存的新内容，数据静默丢失，用户毫无感知。
AI 异步任务会回写 content_json，同时用户在编辑|必须加，否则AI回写会覆盖用户改动
项目有异步任务回写内容的场景，所以 version 字段是有实际价值的——它能防止 AI 任务的回写覆盖用户的手动编辑。

复习 static 和 final

  • static：属于类，不属于对象
    • 全类共享一份
    • 可通过 类名.字段名 访问
  • final：只能赋值一次
    • 基本类型值不可改
    • 引用类型“引用地址不可改”
      组合起来 static final = 类共享 + 不可重新赋值，非常适合常量。
      常量一般声明的时候大写下划线命名

lambdaUpdate() 是 MyBatis-Plus 提供的一个方法，用来构建 基于 Lambda 表达式的更新条件，本质上是一个“链式更新构造器”。

你可以把它理解成：👉 更安全、更优雅地写 UPDATE SQL

lambdaUpdate() 返回的是一个：

👉 LambdaUpdateWrapper<T> 或 LambdaUpdateChainWrapper<T>

用于：

  • 设置更新字段（set()）

Service 类

SseEmitter 是 Spring MVC 里用于 SSE（Server-Sent Events）服务端推送 的类型。
它允许后端保持一个长连接，持续向前端发送事件（比如任务进度、状态变化），而不是前端不断轮询接口。
主要好处是 更实时、更省资源、体验更好：

  • 实时性更高：状态一变化就推送给前端，不用等下一次轮询周期。
  • 减少无效请求：轮询会不断打接口（很多次都是“没变化”），SSE 只在有事件时发数据。
  • 降低后端压力：少了大量重复查询 DB / 缓存的请求，尤其任务多时差别很明显。
  • 前端逻辑更简单：不用管理 setInterval、重试节奏、停止轮询时机等状态机。
  • 网络更友好：长连接 + 增量事件，通常比高频 HTTP 轮询更省带宽与连接开销。
  • 用户体验更自然：进度条、日志、完成提示会“即时跳动”，不会卡在轮询间隔里。
