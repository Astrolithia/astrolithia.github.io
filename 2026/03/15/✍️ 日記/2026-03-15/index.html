八股相关

Redis

特点：高性能、支持大容量读写、读写快访问快、响应时间快、可持久化

可持久化：使用缓存的时候，经常需要对内存中的数据进行持久化，防止机器重启后数据丢失，或者为了数据同步

3种持久化方式：

  1. 快照 RDB
  2. 只追加文件 AOF
  3. RDB 和 AOF 混合持久化

什么是 RDB 持久化
Redis 可以创建快照来获得存储在内存里的数据在某个时间上的副本，Redis 快照后可以对快照进行备份，将快照复制到其他服务器从而创建有相同数据的服务器副本，还可以将快照留在原地以便机器重启时使用

RDB 创建快照时会阻塞主线程吗？
Redis 提供了两个命令来生成 RDB 快照文件

  1. Save：同步保存操作，会阻塞主线程
  2. Bgsave：fork 出一个子进程，子进程执行
    Redis 启动后主要是通过单线程的方式完成主要的工作

Fork 性能开销分析
虽然 bgsave 在子线程中执行，不会阻塞主线程命令请求，但是 fork 操作本身是阻塞的，且会带来额外的内存开销

COW 机制
Fork 后，子进程共享父进程的内存也
当父进程或子进程修改内存页时，内核复制该页
大数据集+高写负载时，会导致大量页面复制，影响性能

THP 透明大页导致的内存雪崩问题
Linux 发行版默认开启 THP,大小为 2 MB。THP 会增加大页被 COW 的概率，最坏的情况下，如果内存被合并为 2 MB 大页，即使客户端仅修改 10 字节的数据，内核也会复制完整的 2 MB 内存页，导致 COW 的内存开销放大 512 倍

AOF 持久化
开启 AOF 持久化后每执行一条会更改 Redis 中数据的命令，Redis 就会将该命令写入到 AOF 缓冲区中，然后再写入到 AOF 文件中，最后再根据持久化方法的配置来决定何时将系统内核缓存区的数据同步到硬盘中

AOF 工作的基本流程是什么样的

  1. 命令追加 所有写命令追加到 AOF 缓存区中
  2. 文件写入 将 AOF 缓存区的数据写到 AOF 文件中
  3. 文件同步 调用 fsync ，针对单个文件操作，对其进行强制硬盘同步，将阻塞直到写入硬盘后返回，保证数据持久化
  4. 文件重写 随着 AOF 文件越来越大，需要定期对 AOF 文件进行重写
  5. 重启加载 当 Redis 重启时，可以加载 AOF 文件进行数据恢复

AOF 为什么在执行完命令之后记录日志
Redis AOP 持久化机制是在执行完命令之后再写记录，为了避免额外的检查开销，AOF 记录日志不会对命令进行语法检查，在命令执行完再记录，不会阻塞当前命令执行；可能会阻塞后续其他命令的执行，如果刚执行完命令 Redis 就出现问题可能会导致对应的修改丢失

什么是 AOP 重写
当 AOP 文件变得太大时，Redis 能够自动在后台重写一个新的 AOP 文件，这个 AOP 和原先的 AOP 文件所保存的数据库状态一样，但体积更小
AOP 重写通过读取数据库中的键值对来实现，程序无需对现有 AOP 文件进行任何读写、分析或者写入
AOP 重写期间，Redis 还会维护一个 AOP 重写缓存区，该缓存区会在子进程创建新 AOP 文件期间，记录服务器执行的所有写命令。当子进程完成创建新 AOP 文件的工作后，服务器会将重写缓存区中的所有内容追加到新 AOP 文件的末尾，使得新 AOP 文件保存到数据库状态与现有的数据库状态一致。最后，服务器用新的 AOP 文件替换旧的 AOP 文件，以此来完成 AOP 文件重写操作。

AOF 校验机制是什么
纯 AOF 模式下，Redis 不会对整个 AOF 文件使用校验，而是通过逐条解析文件中的命令来验证文件的有效性。如果解析过程中发现语法错误，Redis 会终止加载并报错，从而避免错误数据载入内存

Redis-check-AOF 工作原理
根据 AOF 文件逐一读取命令，判断命令参数个数、参数字符串长度等，提供错误/不完整命令的文件位置

人工修复

AOF 文件的组成部分

  1. RDB 快照部分
  2. AOF 增量部分：紧随 RDB 快照部分之后，记录 RDB 快照生成后的增量写命令。

由于 RDB 和 AOF 各有优势，于是，Redis 4.0 开始支持 RDB 和 AOF 的混合持久化
Redis 7.0+开始，混合持久化默认启动
工作原理：AOF 重写的时候就把 RDB 的内容写到 AOF 的开头，这样可以结合 AOF 和 RDB 的优点，快速加载同时避免丢失过多的数据

核心工作流程：

  1. 写处理阶段
  2. 持久化出发阶段
  3. 文件构建阶段
  4. 数据恢复阶段

如何选择 RDB 和 AOF
RDB 比 AOF 优秀的地方：

  1. 文件紧凑，适合备份和灾难恢复：RDB 存储的是经过压缩的二进制数据，保存着某个时间点的数据集，文件很小，非常适合做数据的备份和灾难恢复
  2. 恢复速度快：使用 RDB 文件恢复数据，直接解析还原数据即可，不需要一条条执行命令，速度非常快。而 AOF 则需要依次执行每个写命令，速度非常慢。也就是说，与 AOF 相比，恢复大数据集时，RDB 更快
  3. 主从复制优势：在副本上，RDB 支持重启和故障转移后的部分重新同步。副本可以使用 RDB 快照快速同步到主节点的某个时间点状态，而不需要全量同步
  4. 性能开销小：RDB 最大化 Redis 性能，因为 Redis 父进程需要做的唯一持久化工作就是 fork 子进程，子进程将完成所有其余工作

AOF 比 RDB 优秀的地方：

  1. 数据安全性更高，支持秒级持久化：RDB 的数据安全性不如 AOF,没有办法实时或者秒级持久化数据。生成 RUB 文件的工程是比较繁重的，虽然 BGSAVE 子进程写入 RDB 文件的工作不会阻塞主线程，但会对机器的 CPU 资源和内存资源产生影响，严重的情况下甚至会直接把 Redis 服务干宕机
  2. 版本兼容性好：RDB 文件是以特定的二进制格式保存的，并且在 Redis 版本演进中有多个版本的 RDB,所以存在老版本 Redis 服务不兼容新版本的 RDB 的问题
  3. 可读性和可操作性强：AOF 以一种易于理解和解析的格式包含所有操作的日志。你可以轻松地导出 AOF 文件进行分析，也可以直接操作 AOF 文件来解决一些问题。比如，如果执行
  4. 追加日志无损坏风险：AOF 日志是追加日志，即使日志由于某种原因以半写入命令结尾，也能轻松修复

AOF 和 RDB 的交互
AOF 和 RDB 持久化同时启用时：

  1. 避免同时进行重 I/O 操作
  2. AOF 重写调度：当快照正在进行且用户显示，服务器将返回 OK 状态码，告诉用户操作已调度，重写将在快照完成后开始
  3. 重启恢复优先级：如果 AOF 和 RDB 持久化

3 种常用的缓存读写策略详解

Cache Aside Pattern 旁路缓存模式
最常用、最经典，尤其适合读多写少的业务场景
被称作旁路是因为应用程序的写操作完全绕过了缓存，直接操作数据库

写操作的步骤：

  1. 应用先更新 DB
  2. 然后直接删除 Cache 中对应的数据

读操作的步骤：

  1. 应用先从 Cache 读取数据
  2. 如果命中，就直接返回
  3. 如果未命中，则从 DB 读取数据，成功读取后，将数据写回 Cache,然后返回

为什么是删除而不是更新缓存？

  1. 性能开销：写字段往往只更新了对象的部分字段，如果为了更新缓存而去重新查询或计算缓存对象，开销可能巨大。相比之下，删除是一个轻量级操作
  2. 懒加载思想：删除操作遵循懒加载原则。只有当数据下一次被真正需要时，才从 DB 加载并写入缓存，避免了无效的缓存更新
  3. 并发安全：更新缓存在高并发下可能出现更新顺序错乱的问题导致脏数据的概率会更大

Read/Write Through Pattern 读写穿透
这种模式下，应用程序将缓存视为唯一的、主要的存储。

Write Behind Pattern 异步缓存写入
Read/Write Through 是同步更新 Cache 和 DB,而 Write Behind 则是只更新缓存，不直接更新 DB,而是改为异步批量的方式来更新 DB
写操作：

  1. 应用将数据写入 Cache,然后立即返回
  2. Cache 服务将这个写操作放入一个队列中
  3. 通过一个独立的异步线程/任务，将队列中的写操作批量地、合并地写入 DB

但是，两个又有很大的不同：Read/Write Through 是同步更新 Cache 和 DB,而 Write Behind 则是只更新缓存，不直接更新 DB,而是通过异步批量地方式来更新 DB

写操作：

  1. 应用将数据写入 Cache,然后立即返回
  2. Cache 服务将这个写操作放入一个队伍中
  3. 通过一个独立的异步线程/任务，将队伍中的写操作批量的、合并地写入 DB

这种模式对数据一致性带来了挑战，因此不适用于需要强一致性的场景。但是它的异步和批量特性，带来了无与伦比的写性能，在高性能系统中有广泛应用：比如高频计数场景：对于文章浏览量、帖子点击数这类允许短暂数据不一致、但写入极其频繁的场景,可以先在 Redis 中快速累加，然后再通过定时任务异步同步回数据库

缓存淘汰算法

一个好的缓存淘汰算法应该满足下面这些条件：

  1. 保证缓存命中率高
  2. 维护成本不高，也就是不需要消耗太多资源
  3. 能够应对稀疏流量的场景

稀疏流量指的是在一个时间段内，流量突然出现了大幅增加 但是并不持续，短时间内又恢复正常水平，比如网站某个热门活动开启前后会突然涌入大量的用户访问

缓存淘汰算法

  1. 先进先出算法（FIFO）：按照缓存中数据进入到先后顺序进行淘汰，也就是优先淘汰掉最先缓存的数据，这是最简单一种方式，但是缓存命中率非常低，不推荐使用
  2. 最近最久未使用算法（LRU）：保留最近被使用过的缓存项，淘汰最久未被访问的缓存项。如果一个缓存项长时间没有被访问，那么它很可能之后也不会被访问，因此可以将其淘汰。当系统出现短时间的高并发访问或者突发流量时，很多新的缓存项会被加入，由于缓存空间有限，LRU 可能会将一些访问频率较高的热点缓存项从缓存中淘汰掉
  3. 最少使用算法（LFU）：保留使用频率最高的缓存项，淘汰使用频率最低的缓存项。可以反映访问频率，但其又需要较高的内存占用，而且无法处理稀疏流量的场景
  4. 时钟算法（Clock）：将缓存项看作一个环形链表，每个缓存项有一个引用位，初值都为 0。当某个缓存项被访问时，把它的引用位设置为 1.当需要淘汰缓存项时，从当前位置开始扫描所有缓存项，找到第一个引用位为 0 的缓存项，将其淘汰，无法反映访问频率且对访问模式敏感，该算法可能导致缓存中某些重要的数据被过早淘汰，一般不会采用

Caffeine 采用了优化后的 LFU 和 LRU 的结合——W-TinyLFU 算法，同时具备高命中、低内存占用的特点

W-TinyLFU 算法解决了传统 LFU 算法的两个常见缺陷：

  1. 维护频率信息统计的开销较大
  2. 无法处理稀疏流量的场景
