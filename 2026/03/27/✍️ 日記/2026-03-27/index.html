MySQL 索引

索引是什么？用于快速查找和检索数据的数据结构，其本质可以看成是一种排序好的数据结构
索引的作用就相当于书的目录，查字典的时候如果没有目录只能一页一页翻，如果有目录直接去目录里查找字的位置，然后翻到那一页就可以了
索引的优点：

  1. 查询速度起飞（主要目的）：通过索引，数据库可以大幅减少需要扫描的数据量，直接定位到符合条件的记录，从而显著加快数据检索速度，减少磁盘 I/O 次数
  2. 保证数据唯一性，通过创建唯一索引，可以确保表中的某一列的值是独一无二的，比如用户 ID、邮箱等。主键本身就是一种唯一索引
  3. 加速排序和分组：如果查询中的 ORDER BY 或 GROUP BY 子句涉及的列键有索引，数据库往往可以直接利用索引已经排好序的特性，避免额外的排序操作，从而提升性能

索引的缺点：

  1. 创建和维护耗时：创建索引本身需要时间，特别是对大表操作时，更重要的是，当对表中的数据进行增、删、改时，不仅要操作数据本身

何谓事务？

💡 提示：转账的案例。


事务就是为了解决这个问题：把这两个操作绑定成一个不可分割的整体，要么都成功，要么都不执行。

何谓数据库事务 ？

👉 重要程度： ⭐⭐⭐⭐

💡 提示：数据库事务可以保证多个对数据库的操作（也就是 SQL 语句）构成一个逻辑上的整体。构成这个逻辑上的整体的这些数据库操作遵循：要么全部执行成功,要么全部不执行 。关系型数据库（例如： MySQL 、 SQL Server 、 Oracle 等）事务都有 ACID 特性。

ACID 特性指的是什么？

👉 重要程度： ⭐⭐⭐⭐⭐

💡 提示：ACID 特性大家都知道，但是，这里有一个绝大部分人可能会理解错误的地方：只有保证了事务的持久性、原子性、隔离性之后，一致性才能得到保障。也就是说 A、I、D 是手段，C 是目的！

面试官可能还会顺便问你：“持久性是如何保证的？”、“原子性是如何保证的？”巴拉巴拉。这个时候你就需要结合 MySQL 日志、MySQL 锁以及 MVCC 来回答。

并发事务带来了哪些问题 ?

👉 重要程度： ⭐⭐⭐⭐⭐

💡 提示：脏读（Dirty read）、丢失修改（Lost to modify）、不可重复读（Unrepeatable read）、幻读（Phantom read）。
脏读：
事务 B：UPDATE bill SET amount = 500 WHERE id = 1;（未提交）
事务 A：SELECT amount FROM bill WHERE id = 1; → 读到 500
事务 B：ROLLBACK; → 回滚了
事务 A 读到的 500 是根本不存在的脏数据

丢失修改：
事务 A：读到 amount = 100
事务 B：读到 amount = 100
事务 A：修改为 200，提交
事务 B：修改为 150，提交 → 覆盖了 A 的修改
事务 A 的修改丢失了

不可重复读和幻读区别

👉 重要程度： ⭐⭐⭐⭐⭐

💡 提示：

● 不可重复读的重点是内容修改或者记录减少比如多次读取一条记录发现其中某些记录的值被修改；
● 幻读的重点在于记录新增比如多次执行同一条查询语句（DQL）时，发现查到的记录增加了。

不可重复读：
同一事务内，两次读同一条记录，结果不一样（记录被其他事务修改或删除了）。

幻读：
同一事务内，两次执行同一查询，结果集的行数不一样（其他事务插入了新记录）。

对比 不可重复读 幻读
问题本质 已有记录被修改/删除 新增了记录
针对操作 UPDATE / DELETE INSERT
解决方案 对记录加行锁 需要加间隙锁（Gap Lock）

幻读其实可以看作是不可重复读的一种特殊情况，单独把区分幻读的原因主要是解决幻读和不可重复读的方案不一样。

  • 不可重复读：对已有记录加Record Lock（记录锁） 就能防止被修改
  • 幻读：Record Lock 只能锁住已存在的记录，锁不住还不存在的”间隙”，需要用 Gap Lock（间隙锁） 锁住范围，防止新数据插入

举个例子：执行 delete 和 update 操作的时候，可以直接对记录加锁，保证事务安全。而执行 insert 操作的时候，由于记录锁（Record Lock）只能锁住已经存在的记录，为了避免插入新记录，需要依赖间隙锁（Gap Lock）。也就是说执行 insert 操作的时候需要依赖 Next-Key Lock（Record Lock+Gap Lock） 进行加锁来保证不出现幻读。

SQL 标准定义了哪些事务隔离级别?

👉 重要程度： ⭐⭐⭐⭐⭐

💡 提示：READ-UNCOMMITTED(读取未提交)、READ-COMMITTED(读取已提交)、REPEATABLE-READ(可重复读)、SERIALIZABLE(可串行化)。

MySQL 的默认隔离级别是什么?能解决幻读问题么？

👉 重要程度： ⭐⭐⭐⭐⭐

💡 提示：默认隔离级别是 REPEATABLE-READ（可重读）， 可以解决幻读问题 （两种情况，快照读和当前读）。
快照读：SELECT * FROM bill WHERE month = '2025-03';
通过 MVCC 读取历史版本快照，不管其他事务插入了多少新数据，我读的都是事务开始时的那个版本，天然避免幻读。

当前读：SELECT * FROM bill WHERE month = '2025-03' FOR UPDATE;
读的是最新数据，此时通过 Next-Key Lock 锁住查询范围，其他事务无法在这个范围内插入新记录，避免幻读。

什么是 MVCC？有什么用？原理了解么？

👉 重要程度： ⭐⭐⭐⭐⭐

💡 提示：RC（Read Committed，读提交）和 RR（Repeatable Read，可重复读）这两个隔离级别的实现都离不开 MVCC。

多版本并发控制：通过保存数据的多个历史版本，让读写操作不互相阻塞，读不加锁，提升并发性能。RC 和 RR 两个隔离级别都依赖 MVCC 实现。

表级锁和行级锁了解吗 ？有什么区别？

👉 重要程度： ⭐⭐⭐⭐

💡 提示：表级锁(table-level locking)一锁就锁整张表。行级锁的粒度更小，仅对相关的记录上锁即可（对一行或者多行记录加锁）

行级锁的使用有什么注意事项？

👉 重要程度： ⭐⭐⭐⭐

💡 提示：InnoDB 的行锁是针对索引字段加的锁，表级锁是针对非索引字段加的锁。当我们执行 UPDATE 、 DELETE 语句时，如果 WHERE 条件中字段没有命中索引或者索引失效的话，就会导致扫描全表对表中的所有记录进行加锁。

共享锁和排他锁呢？

👉 重要程度： ⭐⭐⭐⭐

💡 提示：不论是表级锁还是行级锁，都存在共享锁（Share Lock，S 锁）和排他锁（Exclusive Lock，X 锁）这两类

意向锁有什么作用？

👉 重要程度： ⭐⭐⭐⭐

💡 提示：快速判断是否可以对某个表使用表锁。意向锁是表级锁，共有两种。并且，意向锁之间是互相兼容的。

InnoDB 有哪几类行锁？

👉 重要程度： ⭐⭐⭐⭐⭐

💡 提示：记录锁（Record Lock）、间隙锁（Gap Lock）、临键锁（Next-key Lock）。

何为索引 ？有什么作用？

👉 重要程度： ⭐⭐⭐

💡 提示：索引的作用就相当于书的目录。 没有目录，要找某个知识点，只能从第一页翻到最后一页；有了目录，直接定位到页码，秒找到
数据库也一样，没有索引，查数据需要全表扫描，逐行比对；有了索引，可以快速定位到目标数据，大幅减少扫描的数据量
在我的财务平台里，账单表有几十万条数据，没加索引前查询要扫全表，加了索引后直接走 B+ 树定位，查询耗时降低 40%，这就是索引最直观的作用。

索引失效

索引的优缺点

👉 重要程度： ⭐⭐⭐⭐

💡 提示：索引并不都是好的，创建索引和维护索引也需要耗费资源和时间。
优点：

  • 查询速度快：大幅减少查询数据量，提升 SELECT 性能
  • 排序更高效：ORDER BY 字段有索引可以避免 filesort
  • 保证唯一性：唯一索引可以保证数据不重复
    缺点：
  • 占用磁盘空间：索引本身需要额外存储空间
  • 降低写入性能：每次 INSERT/UPDATE/DELETE，都需要同步维护索引结构，有额外开销
  • 索引失效风险：写法不对索引可能失效，反而不如全表扫描

👉 重要程度： ⭐⭐⭐⭐

主键索引和二级索引

👉 重要程度： ⭐⭐⭐⭐

💡 提示：数据表的主键列使用的就是主键索引。二级索引又称为辅助索引，是因为二级索引的叶子节点存储的数据是主键。也就是说，通过二级索引，可以定位主键的位置。唯一索引，普通索引，前缀索引等索引属于二级索引。
主键索引（Primary Key Index）

  • 数据表主键列自动创建
  • 叶子节点存储完整的行数据
  • 一张表只能有一个主键索引
  • 属于聚簇索引

聚集索引与非聚集索引

👉 重要程度： ⭐⭐⭐⭐

💡 提示：聚集索引即索引结构和数据一起存放的索引。主键索引属于聚集索引。非聚集索引即索引结构和数据分开存放的索引。

Redis 单线程模型了解吗？

Redis 单线程指的是处理客户端请求的命令执行是单线程的，
Redis 其他线程负责：

  • 持久化
  • 异步删除大 key
  • 集群数据同步
    IO 多路复用 如果普通阻塞 IO，一个线程只能处理一个请求，那么 1000 个客户端就要开 1000 个线程，线程切换的开销极大
    Redis 的做法：IO 多路复用，用一个线程监听所有客户端的 socket，哪个 socket 有数据来了或者可以写了，就把它放进事件队列，单线程依次从队列里取出来处理

单线程有没有缺点？

  • 慢命令会阻塞整个服务 比如 keys *遍历所有 key 如果数据量大就会导致 redis 卡住，其他命令全部等待
  • 实际的生产环境要禁用 keys * 用 SCAN 替代，分批遍历不会出现阻塞的问题
  • 无法充分利用多核 CPU 只能跑在一个核上 对多核服务器是一种浪费
  • 解决方案：在一台机器上部署多个 Redis 实例

Redis 6.0 之后引入了多线程

Redis 6.0 之后引入了多线程 IO，但只是用多线程来处理网络数据的读写和解析，命令的执行本身依然是单线程。

Redis 6.0 之前为什么不使用多线程？

单线程编程容易并且更容易维护、Redis 的性能瓶颈不在 CPU 。

Redis 给缓存数据设置过期时间有啥用？

  1. 内存是有限的 Redis 数据全在内存中，如果所有数据永久保存，内存迟早被撑爆，设置过期时间让数据自动清理，控制内存占用
  2. 保证缓存数据的时效性 业务数据会变化，缓存不能永久有效，比如我在财务平台里缓存的报表统计数据，如果财务流水出现了更新但缓存没过期，用户看到的就是旧数据，就会出现问题

    设置合理的过期时间，是缓存一致性的最简单兜底方案。

Redis 是如何判断数据是否过期的呢？

Redis 内部维护了一个过期字典，是一个 Hash 表

  • key 是缓存的键名
  • value 是这个 key 的过期时间戳（毫秒）
    每次访问一个 key 时，Redis 会去过期字典里查它的过期时间戳，和当前时间对比，判断是否已过期。

过期的数据的删除策略了解么？

策略一：惰性删除

不主动删，等你来访问的时候再检查。

  • 客户端访问某个 key
  • Redis 先去过期字典查是否过期
  • 过期了就删掉，返回 nil
  • 没过期正常返回
    优点： 对 CPU 友好，不浪费计算资源去扫描
    缺点： 如果某些 key 一直没人访问，就永远不会被删，内存泄漏
    策略二：定期删除

    Redis 每隔一段时间，主动随机抽查一批 key，删掉过期的。

如何保证 Redis 存的都是热点数据？

  1. 设置 maxmemory 限制内存上线
  2. 选用 allkeys-lfu 或 allkeys-lru 淘汰策略
      • LRU：最近最少使用，冷数据自然会被淘汰
      • LFU：按访问频率淘汰，比 LRU 更精准，真正的冷数据会被优先踢掉

怎么保证 Redis 挂掉之后再重启数据可以进行恢复？

依赖持久化机制。Redis 数据全在内存里，一旦宕机或重启，数据全部丢失。而持久化就是把内存数据写到磁盘，重启后从磁盘恢复，保证数据不丢

什么是 RDB 持久化？

  1. RDB 持久化（快照）
    1. RDB 是在某个时间点把内存里的所有数据生成一个二进制快照文件（dump.rdb），类似于给内存拍了一张照片
    2. 是手动触发的方式，如果希望自动触发，可以通过在配置文件中设置规则，比如 save 900 1 就 900 秒内有 1 个 key 被修改就被触发
    3. save 和 bgsave save 主线程执行，会阻塞所有的命令 bgsave 是从主线程 fork 了一个子进程，不阻塞主线程
    4. Fork 子进程 把内存数据写入临时 .rdb 文件 写完后替换旧的 主线程全程不阻塞，继续执行命令
    5. 优点
      1. 文件紧凑体积小，恢复速度快
      2. 对性能影响较小
    6. 缺点
      1. 两次快照之间的数据会丢失，数据丢失风险大
      2. Fork 子进程时如果数据量很大，可能有短暂停顿

什么是 AOF 持久化？

AOF 追加日志 把每一条写命令都追加记录到日志文件里，重启时重放所有命令来恢复数据
客户端发送命令 redis 执行命令 命令写入 aof_buf（内存缓冲区） 根据 appendsync 配置决定如何刷盘

策略 说明 数据安全性 性能
always 每条命令都刷盘 最安全，最多丢1条 最差
everysec 每秒刷一次 ⭐ 最多丢1秒数据 均衡
no 由操作系统决定 最不安全 最好

生产环境一般选 everysec，性能和安全性的最佳平衡点。

AOF 文件会越来越大，Redis 提供 AOF Rewrite 来压缩
比如对同一个 key 执行了 100 次 INCR
AOF 里记录了 100 条命令

重写后只保留最终结果：SET key 100
文件体积大幅缩小

触发重写：BGREWRITEAOF 命令，同样是 fork 子进程处理，不阻塞主线程。
优点

  • 数据丢失风险极低
  • 日志可读

引入混合持久化机制
文件结构：[ RDB 全量数据（某时间点快照）] + [ AOF 增量命令（快照之后的写操作）]

优点：

  • 恢复速度接近 RDB（大部分数据走快照恢复）
  • 数据丢失接近 AOF（只丢增量部分）
  • 两者的优点结合，缺点互补

如何使用 Redis 事务？

👉 重要程度： ⭐⭐⭐⭐

💡 提示：Redis 可以通过 MULTI ， EXEC ， DISCARD 和 WATCH 等命令来实现事务(transaction)功能。
涉及 4 个核心命令 MULTI EXEC DISCARD 和 WATCH

执行 MULTI 之后，所有命令都不会立即执行，而是放进一个命令队列，等 EXEC 时批量执行。

WATCH 实现的是乐观锁思想：

  • 不加锁，只是监听
  • 提交时检查 key 有没有被改过
  • 改过了就放弃事务，由业务层决定是否重试

Redis 事务支持原子性吗？

👉 重要程度： ⭐⭐⭐⭐

💡 提示：Redis 事务在运行错误的情况下，除了执行过程中出现错误的命令外，其他命令都能正常执行。并且，Redis 是不支持回滚（roll back）操作的。因此，Redis 事务其实是不满足原子性的（而且不满足持久性）。

Redis 事务有两种出错情况，表现完全不同：

  1. 语法错误（入队时就报错）
  2. 运行时错误（执行时才出错）

这就是问题所在：Redis 不支持回滚！
出错的命令跳过了，但其他命令照常执行，数据可能处于中间状态，不满足原子性。

Redis 事务还有什么缺陷？

👉 重要程度： ⭐⭐⭐⭐

💡 提示：除了不满足原子性之外，事务中的每条命令都会与 Redis 服务器进行网络交互，这是比较浪费资源的行为。明明一次批量执行多个命令就可以了，这种操作实在是看不懂。 5 条命令就要 5 次网络往返，如果命令多了网络开销非常大，完全没必要。

如何解决 Redis 事务的缺陷？

👉 重要程度： ⭐⭐⭐⭐

💡 提示：Lua 脚本、 Redis functions 。

Lua 脚本的优势

1. 原子性有保证 Redis 执行 Lua 脚本是原子的，脚本执行期间不会被其他命令插入，要么全部执行，要么不执行。
2. 一次网络请求搞定 整个脚本一次性发给 Redis，只有一次网络往返，彻底解决多次网络交互的问题。
3. 支持复杂逻辑 Lua 脚本里可以写条件判断、循环等逻辑，普通事务做不到。
整个判断+修改操作是原子的，不会出现并发问题。

Redis Functions（了解即可）

  • 脚本可以持久化存储在 Redis 里
  • 支持库的概念，复用性更好
  • 实际生产中目前还是 Lua 脚本用得更多

什么是原子性

原子性指的是一个操作或一组操作，要么全部执行成功，要么全部不执行，不存在中间状态。

类比生活中的例子：

  • 银行转账：A 扣款 100，B 收款 100
  • 这两步必须同时成功或同时失败
  • 不能出现”A 扣了钱，B 没收到”的中间状态

如何避免大量 key 集中过期？

👉 重要程度： ⭐⭐⭐⭐⭐

💡 提示：给 key 设置随机过期时间 + 开启 lazy-free（惰性删除/延迟释放）。
比如在我的财务平台里，每天 0 点批量生成当天的报表缓存，如果所有 key 都设置 24 小时过期，第二天 0 点就会同时过期。Redis 定期删除任务需要处理大量过期 key，导致主线程卡顿，同时大量请求打到数据库，可能引发缓存雪崩。

方案一：设置随机过期时间

把过期时间打散，避免同一时刻大量 key 同时失效。

方案二：开启 lazy-free（惰性删除）

开启后，过期 key 的内存释放交给后台线程异步处理，不阻塞主线程，即使集中过期也不会卡顿。

什么是 Redis 内存碎片?为什么会有 Redis 内存碎片?

👉 重要程度： ⭐⭐⭐⭐

💡 提示：内存碎片简单地理解为那些不可用的空闲内存。

内存碎片是指已分配但无法被有效利用的空闲内存。
类比：你有一张 10 格的停车场，停了 4 辆车，但因为车辆大小不一、位置零散，剩下的 6 个格子虽然空着，却放不下新来的大车——这些”空着但用不了”的格子就是碎片。
原因一：内存分配器的机制
Redis 使用 jemalloc 等内存分配器，分配内存时按固定大小的块来分（如 8、16、32、64 字节…），实际需要的大小和分配的大小之间有差距，多出来的部分就是碎片。
原因二：数据频繁修改
原来：key 存了一个 100 字节的 value，占了 128 字节的内存块
更新：value 改成 50 字节，内存块还是 128 字节，多余的 78 字节就是碎片
原因三：大量 key 删除
删除 key 后内存释放，但这些零散的空闲内存可能无法被新数据有效利用。
如何查看内存碎片率？
redis-cli info memory | grep mem_fragmentation_ratio

  • 碎片率 > 1.5：碎片严重，需要处理 -
  • 碎片率 1~1.5：正常范围 -

持久化

Redis 是基于内存的数据库，所以如果没有持久化，那么服务器一旦重启或者断电，之前的所有数据都会丢失
主要有两种方式

  1. RDB(Redis Database)：在一定时间间隔内，将内存中的数据快照写入磁盘，它是某个时间点上数据的完整副本，可以通过配置文件中的 save 参数来配置
    可以根据服务器的负载情况来调整配置
    可以通过 save 命令保存快照
    ls -ltr 列出当前目录下的文件的详细信息
    通过 homebrew 安装的 Redis 的配置文件默认在 /opt/homebrew/etc/

    通过 homebrew 安装的 redis 保存的快照文件在 /opt/homebrew/var/db/redis/ 目录下 使用 xxd dump.rdb 可以查看 2 进制或者 16 进制文件内容的 Linux 命令，就可以看到 rdb 文件中的内容

    快照有一个缺点，那就是如果服务器在快照之后宕机了，那么最后一次快照之后的所有修改内容都会丢失，所以 RDB 更适合用来做备份
    还有我们为 Redis 开辟的内存区域一般都比较大，那么内存中的数据同步到硬盘这个过程就会持续比较长的时间，这段时间 Redis 都是处于阻塞状态，无法接受任何命令

  2. AOF(Append Only File)，意思就是只追加文件，在执行写命令的时候，不仅会将命令写入到内存中，还会将命令写到一个追加的文件中，这个文件就是 AOF 文件，会以日志的形式来记录每一个写操作，当 Redis 重启时就会通过重新执行 AOF 文件中的命令来在内存中重建整个数据库的内容，开启 AOF 只需要在配置文件中将 appendonly 这个参数的值改为 yes 就可以了

缓存击穿

什么是缓存击穿

缓存击穿中，请求的 key 对应的是热点数据，该数据存在于数据库中，但不存在于缓存中（通常是因为缓存中的那份数据已经过期）。这就导致瞬时大量的请求直接打到了数据库上，对数据库造成了巨大的压力，可能直接就被这么多请求弄宕机了

有哪些解决方法？
  1. 永不过期（不推荐）：设置热点数据永不过期或者过期时间比较长
  2. 提前预热（推荐）：针对热点数据提前预热，将其存入缓存中并设置合理的过期时间比如秒杀场景下的数据在秒杀结束之前不过期
  3. 加锁（看情况）：在缓存失效后，通过设置互斥锁确保只有一个请求去查询数据库并更新缓存
缓存穿透和缓存击穿有什么区别

缓存穿透中，请求的 key 既不存在于缓存中，也不存在于数据库中
缓存击穿中，请求的 key 对应的是热点数据，该数据存在于数据库中，但不存在于缓存中（通常是因为缓存中的那份数据已经过期）

缓存雪崩

什么是缓存雪崩

缓存在同一时间大面积的失效，导致大量的请求都直接落到了数据库上，对数据库造成了巨大的压力
缓存服务宕机也会导致缓存雪崩现象，导致所有的请求都落到了数据库上

有哪些解决方法？

针对 Redis 服务不可用的情况：

  1. Redis 集群：采用 Redis 集群，避免单机出现问题整个缓存服务都没办法使用。Redis Cluster 和 Redis Sentinel 是两种最常用的 Redis 集群实现方案
  2. 多级缓存：设置多级缓存，例如本地缓存+Redis 缓存的二级缓存组合，当 Redis 缓存出现问题时，还可以从本地缓存中获取到部分数据

针对大量缓存同时失效的情况：

  1. 设置随机失效时间（可选）：为缓存设置随机的失效时间，例如在固定过期时间的基础上加上一个随机值，这样可以避免大量缓存同时到期，从而减少缓存雪崩的风险
  2. 提前预热（推荐）：针对热点数据提前预热，将其存入缓存中并设置合理的过期时间，比如秒杀场景下的数据在秒杀结束之前不过期
  3. 持久缓存策略（看情况）：虽然一般不推荐设置缓存永不过期，但对于某些关键性和变化不频繁的数据，可以考虑这种策略
缓存预热如何实现？
  1. 使用定时任务
  2. 使用消息队列，比如 Kafka，来异步的进行缓存预热，将数据库中的热点数据的主键或者 ID 发送到消息队列中，然后由缓存服务消费消息队列中的数据，根据主键或者 ID 查询数据库并更新缓存

如何保证缓存和数据库数据的一致性

引入缓存主要是为了提升性能、减轻数据库压力，但却是会带来数据不一致的风险。
绝对的一致性往往意味着更高的系统复杂性和性能开销，所以实践中会根据业务场景选择合适的策略，在性能和一致性之间找到一个平衡点 
常用的一种缓存读写策略：旁路缓存模式（CAP）的读写

  1. 读操作：
    1. 先尝试从缓存读取数据
    2. 如果缓存命中，直接返回数据
    3. 如果缓存未命中，从数据库查询数据，将查到的数据放入缓存并返回数据
  2. 写操作：
    1. 先更新数据库
    2. 再直接删除缓存中对应的数据

更新数据库成功，而删除缓存这一步失败的情况的解决方案：

  1. 缓存失效时间（TTL）变短（不推荐：让缓存数据的）：让缓存数据的过期时间变短，这样的话缓存就会从数据库中加载数据
  2. 增加缓存更新重试机制（常用）：如果缓存服务当前不可用导致缓存删除失败的话，就隔一段时间进行重试，重试次数可以自己定。将删除缓存重试的消息投递到消息队列，然后由专门的消费者来重试，直到成功

Caffeine

本地缓存的方案有哪些？

  1. JDK 自带的 HashMap 和 ConcurrentHashMap
    ConcurrentHashMap 可以看作是线程安全版本的 HashMap ，两者都是存放 key/value 形式的键值对。但是，大部分场景不会将这两者当作缓存，因为只提供了缓存的功能，并没有提供其他诸如过期时间之类的功能
  2. Ehcache、Guava Cache、Spring Cache 这三者是使用的比较多的本地缓存框架
    1. Ehcach
    2. 通过 Spring Cache 的注解实现缓存的话，代码看着会很干净和优雅，但是容易出现问题比如缓存穿透、内存溢出
  3. 后起之秀 Caffeine
    1. 相比于 Guava 来说 Caffeine 在各个方面性能都要更加优秀，而且 Guava 和 Caffeine 的使用方式很像，减少学习成本

本地缓存有什么痛点？

本地缓存的优势：低依赖、轻量、简单、成本低
本地缓存的缺陷：

  • 本地缓存应用耦合，对分布式架构支持不友好，比如同一个相同的服务部署在多台机器上的时候，各个服务之前的缓存是无法共享的，因为本地缓存只在当前机器上有
  • 本地缓存容量受服务部署所在的机器限制明显：如果当前系统服务所耗费的内存多，那么本地缓存可用的容量就很少

分布式缓存常见的技术选型方案有哪些

比较老牌同时也是使用的比较多的还是 Memcached 和 Redis。不过，现在基本没有看过还有项目使用 Memcached 来做缓存，都是直接用 Redis。
最开始 Memcached 比较常用，后面由于 Redis 的发展都转向更加强大的 Redis。
有一些大厂开源了类似于 Redis 的分布式高性能 KV 存储数据库，例如腾讯开源的 Tendis，但是现在没有被维护更新了
还有 Dragonfly 完全兼容 Redis 和 Memcached 的 API，号称全世界最快的内存数据库
KeyDB：Redis 的一个高性能分支，专注于多线程、内存效率和高吞吐量

多级缓存

什么是多级缓存？为什么要用？

最常用的多级缓存实现方式：本地缓存+分布式缓存
本地缓存和分布式缓存虽然都属于缓存，但本地缓存的访问速度要远大于分布式缓存，这是因为访问本地缓存不存在额外的网络开销
一般情况下，不建议使用多级缓存，会增加维护负担（需要保证一级缓存和二级缓存的数据一致性）。而且，其实际带来的提升效果对于绝大部分业务场景来说其实并不是很大
适合多级缓存的两种业务场景：

  • 缓存的数据不会频繁修改，比较稳定
  • 数据访问量特别大比如秒杀场景
    多级缓存方案中，第一级缓存（L1）使用本地内存，第二级缓存（L 2）使用分布式缓存
    读取缓存数据的时候，先从 L1 中读取，读取不到的时候再去 L2 读取，这样可以降低 L2 的压力，减少 L2 的读次数，如果 L2 中也没有，再去数据库查询，数据查询成功后再将数据写入到 L1 和 L2 中

多级缓存一致性如何保证？

在多级缓存系统中，保证强一致性成本太高，业界的几个提供多级缓存功能的缓存框架基本都是最终一致性保证。例如可以使用 Redis 的发布/订阅机制、Redis Stream 或者消息队列来确保当一个实例的本地缓存发生变化时，其他实例能够及时更新其本地缓存，以保证缓存一致性
Canal+广播消息

  1. 首先在数据库中进行数据修改
  2. 使用 Canal 监听数据库的变更操作，当检测到数据变化时，出发缓存更新
  3. 对于 Redis 缓存，因为集群中只共享一份数据，所以直接同步缓存即可
  4. 由于本地缓存分别在不同的 JVM 实例中，需要借助广播消息队列（MQ）机制，将更新通知广播到各个业务实例，从而同步本地缓存

为了不让请求的压力直接来到 MySQL，会使用缓存来处理高并发下的请求
本地缓存的缺点：集群中的每个应用里的缓存数据无法同步
本地缓存的优点：性能实在太强了
