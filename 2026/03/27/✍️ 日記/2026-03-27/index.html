事务

事务是可以在多个一次请求中执行多个命令
Redis 中的事务主要是通过 MULTI 和 EXEC
通过 MULTI 开启事务
在 EXEC 以前，所有的命令都会被放入队列，
在 EXEC 命令后，事务开始执行，事务中的任何一个命令执行失败，不影响其他命令的执行
在事务执行的过程中，其他客户端提交的命令请求，并不会被插入到事务的执行命令序列中
incr k1 k1 自增

命令执行失败不会导致其他命令无法执行

持久化

Redis 是基于内存的数据库，所以如果没有持久化，那么服务器一旦重启或者断电，之前的所有数据都会丢失
主要有两种方式

  1. RDB(Redis Database)：在一定时间间隔内，将内存中的数据快照写入磁盘，它是某个时间点上数据的完整副本，可以通过配置文件中的 save 参数来配置
    可以根据服务器的负载情况来调整配置
    可以通过 save 命令保存快照
    ls -ltr 列出当前目录下的文件的详细信息
    通过 homebrew 安装的 Redis 的配置文件默认在 /opt/homebrew/etc/

    通过 homebrew 安装的 redis 保存的快照文件在 /opt/homebrew/var/db/redis/ 目录下 使用 xxd dump.rdb 可以查看 2 进制或者 16 进制文件内容的 Linux 命令，就可以看到 rdb 文件中的内容

    快照有一个缺点，那就是如果服务器在快照之后宕机了，那么最后一次快照之后的所有修改内容都会丢失，所以 RDB 更适合用来做备份
    还有我们为 Redis 开辟的内存区域一般都比较大，那么内存中的数据同步到硬盘这个过程就会持续比较长的时间，这段时间 Redis 都是处于阻塞状态，无法接受任何命令
    为了处理这种情况，Redis 提供了 bgsave 命令，这样主进程就可以继续处理请求了，但是这个过程中还会有一定的性能损耗，因为 fork 一个子进程也是需要时间的，而在 fork 的过程中 Redis 依然无法处理请求，为了处理这种情况，Redis 又提供了另一种叫做 AOF 的持久化方式
  2. AOF(Append Only File)，意思就是只追加文件，在执行写命令的时候，不仅会将命令写入到内存中，还会将命令写到一个追加的文件中，这个文件就是 AOF 文件，会以日志的形式来记录每一个写操作，当 Redis 重启时就会通过重新执行 AOF 文件中的命令来在内存中重建整个数据库的内容，开启 AOF 只需要在配置文件中将 appendonly 这个参数的值改为 yes 就可以了

在 Vim 模式下，可以通过 /your_keyword 的方式快速搜索关键词

主从复制

一主两从的 Redis 集群，可以实现一定程度上的高可用
主从复制是指将一个 Redis 服务器上的数据复制到其他 Redis 服务器上，也叫主节点和从节点，一个主节点可以有多个从节点，一个从节点只能有一个主节点
一般主节点负责写操作，从节点负责读操作
主节点会将自己的数据异步传输到从节点，从节点接收到主节点的数据之后更新自己的数据，这样就达到数据一致的目的
主节点不需要做任何配置

复制一份 redis 配置文件，如 cp redis.conf redis-6380.conf
透过 code xxx 命令可以用 vscode 打开文件
然后打开配置文件修改 port 为 6380
主进程为了区分开吧 pidfile 加上端口号后缀

把从节点这个配置项也改为 6380

因为要启动多个 Redis 服务，所以要把两个 Redis 的 pid 文件区分开

把 dbfilename 也添加上端口号后缀
dbfilename dump-6379.rdb

dbfilename dump-6380.rdb
当我们执行 save 或者 bgsave 命令后就会把内存中的数据保存到这个文件里面
replicaof 127.0.0.1 6379 表示配置的这个节点是 6379 的从节点

通过 redis.conf 文件开启 redis
redis-server redis-6380.conf

redis-cli -p 6380 指定端口号为 6380
可以通过 info replication 命令查看信息，可以看到这个从节点的角色是 slave，对应的主节点端口号是 6379

可以看到从节点数据和主节点是一样的

Redis 性能优化

大量 key 集中过期问题

对于过期 key，Redis 采用定期删除+惰性/懒汉式删除策略
如果突然遇到大量过期 key 的话，客户端请求必须等待定期清理过期 key 任务线程执行完成，响应速度会比较慢
常见解决方案：

  1. 给 key 设置随机过期时间
  2. 开启 lazy-free（惰性删除/延迟释放），从 Redis 4.0 开始引入，让 Redis 采用异步方式延迟释放 key 使用的内存，将该操作交给单独的子线程处理，避免阻塞主线程
    不管是否开启 lazy-free，都尽量给 key 设置随机过期时间

Redis hotkey

如果一个 key 的访问次数比较多且明显多于其他 key 的话，那这个 key 就可以看作是 hotkey。Hotkey 的出现原因主要是某个热点数据访问量暴增，如重大的热搜事件、参与秒杀的商品

Hotkey 有什么危害

处理 hotkey 会占用大量 CPU 和贷款，会影响

缓存穿透

缓存穿透说简单点就是大量请求的 key 是不合理的，根本不存在于缓存中，也不存在于数据库中。这就导致这些请求直接到了数据库上，根本没有经过缓存这一层，对数据库造成了巨大的压力，可能就直接被这么多请求弄宕机了

解决方法
最基本的就是首先做好参数校验，一些不合法的参数请求直接抛出异常信息返回给客户端。比如查询的数据库 id 不能小于 0、传入的邮箱格式部队的时候直接返回错误消息给客户端等等

  1. 缓存无效 key
    1. 写一个到 Redis 中去并设置过期时间，具体命令如下：SET key value EX 10086。这种方式可以解决请求的 key 变化不频繁的情况，如果黑客恶意攻击，每次构建不同的请求 key，会导致 Redis 中缓存大量无效的 key。
  2. 布隆过滤器
    1. 通过它我们可以非常
  3. 接口限流
    1. 根据用户或者 IP 对接口进行限流，对于异常频繁的访问行为，还可以采取黑名单机制，例如将异常 IP 列入黑名单
    2. 后面提到的缓存击穿和雪崩都可以配合接口限流来解决，毕竟这些问题的关键都是有很多请求落到了数据库上造成数据库压力过大

缓存击穿

什么是缓存击穿

缓存击穿中，请求的 key 对应的是热点数据，该数据存在于数据库中，但不存在于缓存中（通常是因为缓存中的那份数据已经过期）。这就导致瞬时大量的请求直接打到了数据库上，对数据库造成了巨大的压力，可能直接就被这么多请求弄宕机了

有哪些解决方法？
  1. 永不过期（不推荐）：设置热点数据永不过期或者过期时间比较长
  2. 提前预热（推荐）：针对热点数据提前预热，将其存入缓存中并设置合理的过期时间比如秒杀场景下的数据在秒杀结束之前不过期
  3. 加锁（看情况）：在缓存失效后，通过设置互斥锁确保只有一个请求去查询数据库并更新缓存
缓存穿透和缓存击穿有什么区别

缓存穿透中，请求的 key 既不存在于缓存中，也不存在于数据库中
缓存击穿中，请求的 key 对应的是热点数据，该数据存在于数据库中，但不存在于缓存中（通常是因为缓存中的那份数据已经过期）

缓存雪崩

什么是缓存雪崩

缓存在同一时间大面积的失效，导致大量的请求都直接落到了数据库上，对数据库造成了巨大的压力
缓存服务宕机也会导致缓存雪崩现象，导致所有的请求都落到了数据库上

有哪些解决方法？

针对 Redis 服务不可用的情况：

  1. Redis 集群：采用 Redis 集群，避免单机出现问题整个缓存服务都没办法使用。Redis Cluster 和 Redis Sentinel 是两种最常用的 Redis 集群实现方案
  2. 多级缓存：设置多级缓存，例如本地缓存+Redis 缓存的二级缓存组合，当 Redis 缓存出现问题时，还可以从本地缓存中获取到部分数据

针对大量缓存同时失效的情况：

  1. 设置随机失效时间（可选）：为缓存设置随机的失效时间，例如在固定过期时间的基础上加上一个随机值，这样可以避免大量缓存同时到期，从而减少缓存雪崩的风险
  2. 提前预热（推荐）：针对热点数据提前预热，将其存入缓存中并设置合理的过期时间，比如秒杀场景下的数据在秒杀结束之前不过期
  3. 持久缓存策略（看情况）：虽然一般不推荐设置缓存永不过期，但对于某些关键性和变化不频繁的数据，可以考虑这种策略
